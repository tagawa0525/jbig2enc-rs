# Fix: encode_bitimage コンテキストビット取得順序の修正

Status: IMPLEMENTED

## Context

`encode_bitimage`（ジェネリックリージョンの算術符号化）が出力するJBIG2データを `jbig2dec` でデコードすると破損画像になる。all-white画像のみ正常で、黒ピクセルを含む画像はすべて壊れる。

## 原因

`src/arith/image.rs` の `encode_bitimage` において、コンテキストビット c1（y-2行目）と c2（y-1行目）の組み立て順序が C++ 版 `jbig2enc_bitimage()`（`jbig2arith.cc:538-590`）と異なっていた。

### C++ 版（正しい順序）

```cpp
c1 |= (w1 & 0x80000000) >> 31;  // ① w1のMSBを先に取得
// ...
w1 <<= 1;                        // ② その後にw1をシフト
// ...
c1 &= 31;                        // ③ ループ末尾でマスク
```

### Rust 版（修正前・誤った順序）

```rust
c1 = (c1 << 1) & 0x1f;           // シフト+マスク同時
// ...
w1 = w1.wrapping_shl(1);         // ① w1を先にシフト
c1 |= (w1 >> 31) as u16;         // ② シフト後のMSBを取得（1ビットずれ）
```

Rust版はw1/w2のシフトとビット取得の順序が逆のため、コンテキスト値が1ビットずれてしまい、算術符号化の結果が不正になっていた。

all-white画像が偶然動いていたのは、コンテキストに使われるビットもすべて0であるため、取得順序が影響しなかったため。

## 修正内容

C++版と同じ順序に修正:

```rust
// ① 先にw1/w2のMSBをc1/c2に取得
c1 = (c1 << 1) | ((w1 >> 31) as u16);
c2 = (c2 << 1) | ((w2 >> 31) as u16);
c3 = (c3 << 1) | v as u16;

// ② その後にw1/w2をシフト/リロード
if m == 28 && y >= 2 { w1 = ...; } else { w1 <<= 1; }
if m == 27 && y >= 1 { w2 = ...; } else { w2 <<= 1; }
if m == 31 { w3 = ...; } else { w3 <<= 1; }

// ③ マスクはシフト/リロード後（C++版と同じ位置）
c1 &= 0x1f;
c2 &= 0x7f;
c3 &= 0x0f;
```
